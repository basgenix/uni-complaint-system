4. Functional Requirements

4.1 User Management Module

FR-001: User Registration

Priority: High

Description: Students can create accounts using their university email and matriculation number.

Acceptance Criteria:

  • Email must be valid university domain
  • Matric number must be unique
  • Password minimum 8 characters
  • Email verification required

FR-002: User Authentication

Priority: High

Description: Secure login system with JWT token authentication.

Acceptance Criteria:

  • Support email/password login
  • Session timeout after 24 hours
  • Account lockout after 5 failed attempts
  • Password reset via email

4.2 Complaint Management Module

FR-004: Submit Complaint

Priority: High

Description: Students can submit new complaints/requests with full details and attachments.

Acceptance Criteria:

  • Select category from predefined list
  • Enter title (max 200 characters)
  • Enter description (unlimited)
  • Attach files (optional, max 5MB each)
  • Auto-generate unique ticket number
  • Display confirmation message

FR-005: Track Complaint

Priority: High

Description: Real-time tracking of complaint status with detailed history.

Acceptance Criteria:

  • View list of all submitted complaints
  • Filter by status/category/date
  • Search by ticket number
  • View detailed complaint information
  • See complete response history

4.3 Admin Management Module

FR-007: View All Complaints

Priority: High

Description: Admins can view and manage all complaints in the system.

Acceptance Criteria:

  • List view with pagination
  • Advanced filtering by status/category/date/priority
  • Sort by any column
  • Comprehensive search functionality
  • Export to CSV/PDF

FR-009: Update Complaint Status

Priority: High

Description: Admins can update complaint status with proper documentation.

Acceptance Criteria:

  • Status options: Pending, In Progress, Resolved, Closed, Rejected
  • Require comment for status change
  • Automatic notification to student
  • Record timestamp and admin details

6. System Architecture

Three-Tier Architecture

The system follows a modern three-tier architecture pattern:

  • Presentation Layer: React.js with Tailwind CSS for responsive, modern UI
  • Application Layer: Flask REST API handling business logic and data processing
  • Data Layer: PostgreSQL database with SQLAlchemy ORM for data persistence

Key Architectural Components

  • Load Balancer (Nginx): Distributes incoming traffic across multiple servers
  • API Gateway: Central entry point for all API requests with authentication
  • Caching Layer (Redis): Improves performance by caching frequently accessed data
  • File Storage: Secure storage for complaint attachments and documents

7. Database Design

7.1 Core Tables

Users Table

Stores all user information including students, admins, and super admins.

Key Fields: id, email, password_hash, full_name, matric_number, department, faculty, phone, role, is_active, created_at, updated_at

Indexes: email, matric_number, role

Complaints Table

Central table storing all complaint/request information.

Key Fields: id, ticket_number, user_id, category, title, description, status, priority, assigned_to, attachments, created_at, updated_at, resolved_at

Indexes: ticket_number, user_id, status, category, created_at

Responses Table

Stores all responses and communications related to complaints.

Key Fields: id, complaint_id, user_id, message, is_internal, created_at

Relationships: Many responses per complaint

Notifications Table

Manages all system notifications for users.

Key Fields: id, user_id, complaint_id, type, message, is_read, created_at

Purpose: Real-time updates and alerts
